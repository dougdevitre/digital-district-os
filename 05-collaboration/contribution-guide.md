# Contribution Guide

## How to Contribute

This is an open-source project. Contributions are welcome from anyone — city officials, technologists, community organizers, entrepreneurs, researchers, and policy makers.

## Types of Contributions

### 1. Documentation Improvements
- Fix errors, clarify language, update outdated information
- Add examples from your city or corridor
- Translate content into additional languages

### 2. New Case Studies
- Document your city's Digital District deployment in `09-case-studies/`
- Use `09-case-studies/template-case-study.md` as your starting point
- Include honest assessment of what worked and what didn't

### 3. Template Enhancements
- Improve existing templates in `templates/`
- Add templates for common use cases not yet covered
- Localize templates for different regulatory environments

### 4. Decision Tree Updates
- Add decision branches for scenarios not yet covered
- Update criteria based on real-world deployment experience
- Add data points that improve decision quality

### 5. Technical Specifications
- Improve API design standards
- Add new MCP server implementations
- Enhance security specifications
- Contribute interoperability standard extensions

### 6. Messaging and Communications
- Add audience-specific pitches for stakeholders not yet covered
- Update objection handling with new objections encountered in the field
- Translate messaging for different cultural/political contexts

## Process

### For Small Changes (typos, clarifications, single-file updates)
1. Fork the repository
2. Make your change
3. Submit a pull request with a clear description
4. A maintainer will review within 5 business days

### For Large Changes (new modules, structural changes, new standards)
1. Open an issue describing what you want to add/change and why
2. Discuss with maintainers and community
3. Once approach is agreed, fork and build
4. Submit a pull request referencing the issue
5. Review period: 10 business days minimum for substantial changes

### For New Case Studies
1. Use the template
2. Include real data where possible (anonymize if needed)
3. Be honest about failures — anti-patterns are as valuable as success stories
4. Submit via pull request

## Standards

### Writing Style
- Clear, direct, practical
- Headings + tables + actionable items
- No jargon without definition (add to glossary if introducing new terms)
- Cite sources for factual claims
- Use "the Digital District" (not "our Digital District") to keep content portable

### File Naming
- Lowercase, hyphenated: `corridor-selection.md`
- Descriptive: readers should know what the file contains from the name

### Formatting
- Markdown with GitHub-flavored extensions
- Tables for structured comparisons
- Code blocks for technical specifications
- Decision trees as indented text blocks (not images — they're not searchable)

## Code of Conduct

This project exists to help communities build equitable digital infrastructure. Contributions must:

- Respect all communities, especially those historically excluded from innovation ecosystems
- Avoid partisan political framing (this is a practical infrastructure project, not a political statement)
- Use child-safe, trauma-informed language in any content that touches vulnerable populations
- Not promote specific vendors or products (recommend categories and open standards)

## Governance of Contributions

Major changes to core documents (SKILL.md, framework files, standards) require review by at least two maintainers. Community feedback period of 14 days for changes that affect the interoperability standard.

## Recognition

All contributors are recognized in CONTRIBUTORS.md. Substantial contributions (new modules, case studies, significant improvements) are highlighted in release notes.
